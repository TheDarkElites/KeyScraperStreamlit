Quote from: priyadi on Wed, 12 April 2017, 20:04:23
... I don't think it would be able to even emulate the default Planck layout. ...

What do you mean by that?

What aspects of the default Planck layout you are referring to?

The default Planck layout has 7 layers, ps2avr only got 4 layers max. Default Planck support switching default layer between qwerty, dvorak and colemak; AFAIK, ps2avr can't change default layout. Default Planck uses raise+lower to switch to ADJUST layer; I don't think this can be easily accomplished in ps2avr. Default Planck outputs _ and + (shifted state), this would require complicated macro in ps2avr.

Recent QMK has ps2avr support, and with it, it should support the default Planck layout. But why not go all the way and use atmega32u4 instead? We'd get all the features and easy programming using qmk.sized.io.

What aspects of the default Planck layout you are referring to?

The default Planck layout has 7 layers, ps2avr only got 4 layers max. Default Planck support switching default layer between qwerty, dvorak and colemak; AFAIK, ps2avr can't change default layout. Default Planck uses raise+lower to switch to ADJUST layer; I don't think this can be easily accomplished in ps2avr. Default Planck outputs _ and + (shifted state), this would require complicated macro in ps2avr.

Recent QMK has ps2avr support, and with it, it should support the default Planck layout. But why not go all the way and use atmega32u4 instead? We'd get all the features and easy programming using qmk.sized.io.
If Bface bothers you but you like the deisgn of this case, feel free to use the Plank PCB instead.
